# Reflexive

Reflexive entries respond to the SYSTEM's own events. They are self-governance actions, distinct from reactive (which responds to user context).

## Trigger Subtypes

### Lifecycle

Fires on session lifecycle events.

| Trigger | When |
|---|---|
| `"session boot complete"` | After all implicit tiers finish |
| `"session closing"` | Before session ends |

### Mutation

Fires on codex or state changes.

| Trigger | When |
|---|---|
| `"codex module edited"` | Executable files in `.codex/` are modified |

### Temporal

Fires on cron-style schedules. Standard 5-field cron expression.

| Trigger | When |
|---|---|
| `"cron */5 * * * *"` | Every 5 minutes |
| `"cron 0 9 * * 1-5"` | Weekdays at 9am |

Temporal definitions are durable (live in codex). Runtime cron registrations are per-session — generated by `codex-register` at boot, ephemeral, consistent with Claude Code's session-scoped cron lifecycle.

## Platform Correlation

Temporal triggers generate `.claude/` cron registrations via `codex-register`. The codex entry is authoritative; the cron registration is ephemeral.

## Isolation

Any reflexive entry may declare `isolation: subagent` to run in its own context window. When isolated, `reads:`/`writes:` constrain access.

Note: reflexive triggers fire automatically. The isolation decision should account for the trigger's frequency and the cost of context window allocation. A mutation trigger that fires on every edit should likely remain `inline`; a session-closing compliance check may benefit from `subagent` isolation to avoid polluting the main conversation.
